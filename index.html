Part 01
01
AI 为什么总像"第一天上班"？
根本原因 · 无状态

它很聪明，可惜没有记性

  • →每次请求只看得到当前 context window，对话一关就清零
  • →所谓"它还记得"，只是客户端每次把整段历史重新发了一遍
  • →长上下文 ≠ 长期记忆：贵（每轮重复付费）· 慢且中间信息易被忽略（lost in the middle）· 关了还是没了，解决不了跨会话
真正的记忆 = 持久化存储 + 精准召回，是模型之外的两个能力。
真实例子 你：明天天气好的话能干嘛，要不要去遛狗？ Claude：既然你养了蛋蛋（你的金毛边牧），它运动量很大，可以带它散个长步、玩飞盘，或者骑车带它一起跑…
整段对话里从没提过这只狗。这条信息是从很久以前的另一次会话里召回出来的。
记忆的三个价值
  • 01省去重复交代
  • 02个性化的连续性
  • 03跨时间积累的工程上下文代码风格、踩过的坑、技术决策的"为什么"

对照 · Claude 自己怎么记

Claude 自己，是怎么记的？

持久 · 写在磁盘
  • CLAUDE.md 指令managed / user / project / local 四级：~/.claude/CLAUDE.md、./CLAUDE.md
  • Auto memory 笔记~/.claude/projects/<repo>/memory/：MEMORY.md 索引 + topic 文件，per repo、机器本地
工作 · 会话内易失
  • Context window约 200K tokens，会话结束即清空
  • Auto-compact 压缩接近上限时写摘要保留；CLAUDE.md 重新注入，临时指令会丢

另有两套别混淆：API memory tool（工具式，存储你自己实现）与 claude.ai 账号记忆（云端托管）。两者都不是 Claude Code 的本地记忆。

共性与局限

都挺好，只是不归你

维度厂商记忆
归属锁死在单一产品内，记忆出不了那个 App
透明度黑盒，难以逐条查看 / 编辑 / 精确删除
可移植不跨工具，更不跨 CLI
数据主权存在厂商服务器上
可控性存什么、召回什么，用户基本说了不算

面向"产品用户"设计，不是面向"在终端里干活的开发者"设计。

Part 04
04
它到底是怎么记住的？
整体 pipeline

三个 Hook，在你没察觉时记住一切

提交 prompt 时

UserPromptSubmit

embed 你的输入 → 召回 Top-K 相似历史 → 注入 context（有摘要则注入摘要）

一轮结束时

Stop

落库 SQLite + Chroma，并 fork 一个异步 summarizer 提炼长期记忆

会话结束时

SessionEnd

对整段会话做一次粗粒度摘要，相当于"会议纪要"

Prompt→ Embed→ Recall→ Inject↔ Persist→ Summarize 零特殊语法，全自动，对用户无感
存储层

记忆到底存在哪

真相源 source of truth

SQLite

原始 turns、每轮摘要、session 摘要、tags，结构化、可精确查询

本地向量索引

Chroma

turns + summaries 的向量，负责语义相似检索

Embeddings

Voyage AI voyage-3.5

把文字变向量

Summarization

OpenAI / DeepSeek / Qwen

配哪个 key 用哪个，auto 自动选，不被单一厂商绑死

三个关键设计

好不好用，就看这三点

隔离

双层存储

project 与 global 两层，按 .lynx-memory/ 标记自动切换，可互相 merge

信噪比

Goal 目标门控

入库前 LLM 判定相关性，不相关不入库。strict + fail-open：出错也不丢数据

可移植

跨工具 + 跨机器

Claude Code 与 Codex 共享一份；Turso embedded replica 按 git remote 绑定，换机器照样召回

本地优先：读写先打本地文件，改动后台异步同步，连向量一起传，换机器不用重新 embedding。

外围能力

你的记忆，能打开看、能管理

  • ▸Web UIFastAPI + React，只绑 127.0.0.1。卡片画廊 + 详情抽屉，关键词 / 语义搜索，打标，看 Retrievals，中英切换，⚙ 设置
  • ▸Daily digest每日小结，可推送到微信（ServerChan）
  • ▸MCP serversearch_memory / list_recent / stats / forget

痛点OpenLynx 的解
